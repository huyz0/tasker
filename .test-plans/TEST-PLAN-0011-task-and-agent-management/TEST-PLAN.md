# Task and Agent Validation

## Context & Objective
Tasker is fundamentally a system interacting with humans and intelligent agents through defined tasks. This test plan validates the data persistency, API accuracy, and Event-Driven propagation of Tasks and Agents, ensuring strict validation to avoid system failure due to corrupted input payloads generated by hallucinating LLMs.

## Scope of Testing
### In Scope
- Validation of `tasks` Drizzle inserts + relational bindings.
- Validation of `agents` and `agent_roles` instantiation.
- Task to Agent Assignment persistence via Junction Tables.
- Backend REST/RPC route implementations testing.
- Event pub/sub assertions for NATS (TaskCreated, AgentCreated).

### Out of Scope
- Custom UI test automation (Task-to-Artifact linkers, Custom JSON schema builders).
- Front-end query proxy functionality integration.

## Test Strategy & Environment
- **Unit/Integration**: Handlers within `/src/modules/tasks` and `/src/modules/agents` require Drizzle local container mock testing with SQLite/Vitest to ensure CRUD operability, specifically focusing on validating Zod input parsing accuracy.
- **E2E Critical Path**: Since standard UIs or API calls will be leveraged, direct `.http` payload submission checking via RPC logic represents the vital E2E equivalent.
- **Data/Mocking Needs**: A `test_organization`, `test_user`, and `test_project` must exist within the environment prior to seed tasks. A mock NATS handler must observe publish operations.

## Test Cases

### TC-001: Task Creation (Happy Path)
- **Type**: Integration
- **Given**: A valid `projectId` and valid textual `title/description`.
- **When**: The user calls `TaskService.createTask`.
- **Then**: Record is persisted in the Drizzle SQLite instance and returned flawlessly; a NATS `TaskCreated` event is fired.

### TC-002: Agent Role and Instance Creation
- **Type**: Integration
- **Given**: Valid parameters specifying a System Prompt and capabilities.
- **When**: A request hits `AgentService.createAgent` specifying an `agentRoleId`.
- **Then**: Agent successfully attaches relationally to Organization and Role.

### TC-003: Task Assignment Configuration
- **Type**: Integration
- **Given**: Existing `taskId` and `agentId`.
- **When**: The Assignment logic is invoked mapping the Task.
- **Then**: The record persists into the `task_assignments` DB junction, ensuring unique compound constraint.

### TC-004: Zod Payload Rejection (Hallucination Defense)
- **Type**: Unit
- **Given**: A badly structured payload lacking required identifiers or containing rogue path injections.
- **When**: Submitted to the routing payload verifier.
- **Then**: RPC strictly throws an immediate `400 Bad Request` prior to hitting ORM queries.

### TC-005: NATS Event Firing Contract
- **Type**: Integration
- **Given**: Database commit has successfully concluded for a Task.
- **When**: The pub/sub utility is executed.
- **Then**: A listener scoped to `system.tasks.events` intercepts the cleanly parsed schema object.
